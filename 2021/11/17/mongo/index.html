MongoDB란?

NoSQL과 mongoDB

mongoDB, Replication과 Sharding

NoSQL

NoSQL이란 Not Only SQL의 약자이다. 전통적인 RDB 관계형 데이터 베이스보다 덜 제한적인 일관성 모델을 제공하는 DB를 의미한다. 기존의 RDB 보다 좀 더 유연하고, 확장성이 있다는 차이점을 가지고 있다.

기존의 RDBMS, 대표적으로 MySQL, PostgreSQL, Oracle DB가 있으며 모두 1970~1990년대에 탄생되었다.

새롭게 등장한 NoSQL DBMS는 대표적으로 mongoDB, elasticsearch, redis등이 있으며 대략 2009년~2010년 사이에 탄생되었다.

수십년간 안정적이고 대중적으로 사용되던 기존의 RDBMS에서 벗어나 NoSQL DBMS가 탄생되게 된 배경은 무엇일까?

그것은 바로 스마트폰의 등장과 인터넷의 확산에 배경이 있다.

스마트폰이 등장한 것 뿐 아니라 기존에 주고 받는 데이터의 형식 또한 정형화된 데이터 뿐만 아니라 굉장히 다양한 데이터 형태가 생겨나게 되면서 기존의 데이터 베이스로는 해결하기 힘든 시점에 도달하였고, 이를 극복하기 위해서 NoSQL이 탄생하게 되었다.

RDBMS와 NoSQL DBMS

RDBMS의 기본적인 규칙들은 다음과 같은데 이러한 규칙들로 하여금 RDBMS를 보다 안정성 있게 만들어 주었다.

  • ACID 원칙을 준수
  • 2차원의 테이블 형태
  • SQL을 통한 쿼리

그러나 서버에 요구되는 정보 처리량이 급속하게 증가하면서 처리 속도에 대한 필요성이 대두되었고 이러한 일부 규칙들을 포기하면서 확장성과 성능을 최적화한 것이 NoSQL DBMS이다.

때문에 NoSQL DB는 확장성, 유연성, 고성능, 고기능성이라는 장점을 갖는다.

  • 확장성 : 하나의 고성능의 DB서버 대신, 분산형 클러스터 컴퓨팅을 활용하여 확장성 있도록 설계 되었음.
  • 유연성 : SQL보다 유연한 스키마를 사용하여 빠르고 유연하게 데이터를 저장할 수 있음.
  • 고성능 : 특정 데이터 모델, 엑세스 패턴에 최적화되어 뛰어난 성능을 얻게 해줌
  • 고기능성 : 각 데이터의 모델에 맞게 구축된 뛰어난 기능의 API와 데이터 유형을 제공


MongoDB

mongoDB는 앞에서 언급했듯이 NoSQL 기반이며 꾸준한 인기를 얻고 있는 DB이다.

2021년 stack over flow survey에 따르면 세번째, 매우 높은 순위로 개발자들에게 사랑받고 있는 DB임을 알 수 있다. 놀라운 것은 상위권에 해당하는 DB역시 대부분 NoSQL 인 것을 확인할 수 있다.

개발자들에게 가장 선호되는 데이터베이스는 PostgreSQL에 살짝 뒤쳐지지만 그래도 근소한 차이로 2위를 차지할 만큼 인기가 높다.

또한 MongoDB는 카카오, 등 많은 곳에 활용되며, 기존의 RDBS와는 다르게 분산컴퓨팅을 기본적으로 지원한다는 특징을 가지고 있다.

MongoDB 특징

mongodb의 특징은 다음과 같다.

  • Reliability(신뢰성)
    • 서버 장애에도 서비스가 계속 동작될 수 있음을 의미
    • Primary와 Secondary로 구성된 ReplicaSet 으로 고가용성을 지원함
  • Scalability(확장성)
    • 데이터와 트래픽 증가에 따라 scale-out 확장 가능함을 의미
    • 데이터를 샤딩하여 scale-out이 가능함
  • Flexibility(유연성)
    • 여러가지 형태의 데이터를 손쉽게 저장가능함을 의미
    • NoSQL의 대표적인 장점으로 스키마 변경 없이 요구사항에 맞춰 다양한 형태의 데이터 형식을 저장하고 읽을 수 있음
  • Index-Support(인덱스 지원)
    • 다양한 조건으로 데이터를 빠르게 조회할 수 있음.
    • 여러가지 형태의 인덱스를 지원, 다양한 용도로 사용 가능함.

MongoDB와 분산 컴퓨팅

MongoDB에서 지원하는 분산컴퓨팅 기능으로는 Replication, Sharding이 존재한다.

Replication(복사하여 저장)

A replica set in MongoDB is a group of mongod processes that maintain the same data set.

MongoDB 공식문서에 따르면 replica set이 중복성과 고가용성을 제공한다고 한다.

여러개의 데이터 복제본을 다른 데이터 베이스 서버에 둠으로써 한 개의 데이터 베이스 서버에서 발생할 수 있는 데이터 손실 문제에 있어 이점이 있다.

또한 경우에 따라 여러 데이터를 복제해 둠으로써 read 요청 수용량을 늘릴 수 있고, 데이터의 locality와 availablitiy를 늘릴 수 있다고 한다. 또한 목적에 따라 재해, 백업 용도로 데이터를 복제할 수 있다.

이는 곧 데이터의 안정성, 고가용성을 높일 수 있다는 의미를 가지며, 앞서 언급했던 특징 중 Reliability(신뢰성)에 해당하는 기능이다.

Sharding(나누어 저장)

Sharding is a method for distributing data across multiple machines.

MongoDB의 Replica set의 데이터가 증가하여 하나에 담을 수 없을 경우 데이터를 샤딩하여 분산시켜 저장할 수 있다.

또한 특정 샤드에 데이터가 몰리면 다른 샤드로 데이터를 이동시켜 전반적으로 모든 샤드가 균등하게 데이터를 저장할 수 있도록하는 밸런싱 기능도 있다.

Shard Cluster의 구성요소와 작동방식

샤드 클러스터는 다음과 같이 3가지 구성요소로 구성됨.

  • shard: Each shard contains a subset of the sharded data. Each shard can be deployed as a replica set.
  • mongos: The mongos acts as a query router, providing an interface between client applications and the sharded cluster. Starting in MongoDB 4.4, mongos can support hedged reads to minimize latencies.
  • config servers: Config servers store metadata and configuration settings for the cluster.

샤드 클러스터에 저장되는 실제 데이터는 shard에 나뉘어서 저장된다. 그리고 어떤 데이터가 어떤 shard에 저장되어 있는지 확인할 수 있는 메타데이터와 구성 설정은 config server에 저장된다.

Application server는 mongos서버를 통해 샤드 클러스터 데이터에 엑세스하는데 이때, config server와 통신하여 요청받은 데이터가 어떤 샤드에 들어있는지 파악한 뒤 해당 샤드에서 데이터를 조회하여 application server에 제공하게 된다.

Sharding의 장점

읽기/쓰기

  • 샤딩은 성능을 향상하기 위한 방식으로, 읽기와 쓰기 작업을 샤드클러스터의 shard에 분산하여 각 샤드가 이를 처리할 수 있도록 한다. 또한 샤드를 추가하는 수평적으로 추가하는 방식으로 읽기와 쓰기 작업을 확장시킬 수 있다고 한다.
