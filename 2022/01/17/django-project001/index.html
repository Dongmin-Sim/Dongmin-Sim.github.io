이어드림 과정을 수강하면서 사이드 프로젝트를 진행하게 되었다. 사이드 프로젝트의 백엔드는 django로 구현할 예정이며 이를 위해 django 학습 목적으로 본 글을 작성한다. django doc을 참고하여 작성하였음을 미리 밝힌다.


django 프로젝트 생성

장고 프로젝트를 생성하기 위해서는 다음과 같은 명령어를 사용해주면 된다. 프로젝트의 이름만 설정해주면 그 다음부터는 django가 알아서 자동으로 기본 코드를 생성해준다. 여기에는 데이터베이스 설정, 세부 옵션, app-설정을 포함한 django instance를 위한 설정들이 포함된다.

django에서는 프로젝트의 이름을 build-in python,혹은 django Component의 이름을 피할 것을 권장한다.
예시로 django라던가, test와 같은 naming은 충돌을 일으킬 수 있다.


django 프로젝트 구조

만일 django 프로젝트를 생성할 때 프로젝트 이름을 coco로 생성했을 때, 만나게될 프로젝트 구조는 다음과 같다.

root directory

  • coco/는 우리 프로젝트의 root directory가 된다. 이 root directory의 이름은 django에 영향을 끼치지 않아서, 원하는 directory 이름으로 바꾸어도 무방하다.
  • manage.py 파일은 django와 상호작용할 수 있는 command-line utility 파일이다.
  • root directory안에 있는 또 다른 coco/ directory가 프로젝트를 위한 실제 파이썬 코드들이 있는 공간이다. 이곳을 project directory라고 부르겠다. 파이썬 패키지 name으로 해당 디렉토리안에 있는 파일들을 import 할 수 있다 (ex. coco.urls)

project directory

  • coco/__init__.py은 빈 파일이다. 위에서 언급했듯이 해당 디렉토리를 파이썬 패키기로 묶기 위해 생성된 파일이다.

  • coco/settings.py django project를 위한 세팅,설정을 위한 파일이다. django_settings

  • coco/urls.py 프로젝트를 위한 urls이 선언되는 파일이다. 우리의 django 사이트를 위한 목차 역할을 한다.

  • coco/asgi.py 프로젝트에 사용할 ASGI 호환 웹 서버의 진입점이다.

    • asgi란 무엇일까, 웹서버,프레임웤, 응용 프로그램간 호환성을 담당하는 wsgi의 정신적 계승자라고 한다. wsgi는 요청을 받고, 응답을 반환하는 단일 동기 호출 기능을 하는데, 이는 복잡한 웹 프로토콜을 처리하는데 있어서 적합하지 않다고 한다. 그래서 나온 asgi는 비동기&동기 기능을 제공하며, 다양한 서버와 동기 기능의 표준을 제공해 준다고 한다. asgi/wsgi에 관해서 따로 정리가 필요하다.

  • coco/wsgi.py 프로젝트에 사용할 WSGI 호환 웹 서버의 진입점이다.

    • wsgi는 web server gateway interface라고 하며, web server와 web application과 대화하게 해줄 수 있는 하나의 인터페이스이다. 이에 대한 좋은 글이 있어 링크를 남겨둔다.wsgi란?

정리하자면 django-admin으로 django 프로젝트를 생성하게 되면, naming한 프로젝트 이름으로 루트 디렉토리가 생성되고, 루트 디렉토리 안에는 manage.py파일과 프로젝트 이름의 프로젝트 디렉토리가 생성된다.

django에서 작성된 어플리케이션들은 특정한 컨벤션을 따르는 파이썬 패키지로 구성되어 있다.
django에서는 자동적으로 어플리케이션의 기본 구조를 생성해주기 때문에 따로 directory를 만들 필요가 없다.

django 프로젝트와 앱은 무엇이 다른가?

장고 앱은 웹 어플리케이션으로, 게시판 앱, 문의사항 앱 등처럼 웹 페이지에서의 기능적(?)단위라고 볼 수 있으며, 프로젝트는 특정 웹사이트를 위한 설정과 앱들의 콜렉션이라고 볼 수 있다. 따라서 프로젝트는 여러개의 앱들을 포함할 수 있다. 구현하고자 하는 웹페이지를 프로젝트라고 한다면, 안에 들어가는 기능이나, 구성을 앱으로 구현할 수 있다.

django MVT 구조

django는 MVT 디자인 패턴을 갖는다. 나는 디자인 패턴을 소프트웨어를 설계할 때 발생되는 문제들을 최소화하기 위해 이를 해결할 수 있는 다양한 유사한 방법들(패턴)을 재사용하기 좋은 방식으로 정리한 방법론이라고 이해를 했다. 정의된 디자인 패턴은 효율적인 의사소통의 수단으로써 사용될 수도 있다.

카카오 콩을 정제한 카카오매스에 설탕 등을 첨가해 만든 달콤한 맛의 과자를 먹자 == 초콜릿(디자인 패턴)을 먹자

나중에 추후에 디자인 패턴에 대해서도 정리해봐야겠다. 소프트웨어를 효율적으로 설계하기 위한 구조적 특징을 패턴화 한 것으로 이해하면 좋을 것 같다. 이러한 디자인 패턴은 이외에도 매우 많은 방식의 패턴들이 존재한다.

그 중에서도 django는 MVT 디자인 패턴을 갖는다. MVT란 Model-View-Template를 의미한다. MVC(Model-View-Controller)와 같은 계열의 패턴이라고 볼 수 있을 것 같다.

  • Model :

    • DB와 관련된 부분이며, DB에 저장되는 데이터를 의미한다.
    • SQL에서의 TABLE을 클래스로 정의한다.
    • 모델 클래스에서 저장될 데이터의 속성을 정의하고, 이를 기반으로 데이터의 CRUD가 가능하다. django 또한 flask의 sqlalchemy처럼 ORM을 이용하여 파이썬 코드로 DB를 조작하는 것이 가능하다. 물론 SQL문을 사용해야하는 경우도 종종 있다고 한다. (데이터와 관련된 필요한 기능을 최대한 클래스에서 함수로 짜라는 이야기를 들은 것 같다..)

  • View :

    • MVC 패턴에서의 Controller의 역할을 한다.
    • 요청을 받아 해당 요청을 잘 처리할 수 있는 로직을 담당한다.
    • template과 model에 연관되어, 응답으로 템플릿 렌더링 혹은 데이터를 전달해주는 기능을 할 수 있다.

  • Template :

    • View에서의 로직의 결과로 유저에게 화면을 보여주는 역할을 한다.
    • html 파일이 이에 해당되며 django template 문법으로 보다 편리하게 데이터와 함께 렌더링할 수 있다.

위의 그림처럼 include() 함수는 다른 URLconf를 참조할 수 있도록 도와준다. django는 위의 그림과 같이include()를 만나게 되면, URL의 그 시점까지 일치하는 부분을 잘라내고,다음 부분 문자열을 해당하는 include된 URLconf로 전달한다. include는 url을 쉽게 연결해 줄 수 있는 함수이다. include()함수는 다른 URL패턴을 포함할 때마다 항상 사용하여야 하며 관리자 페이지로 연결되는 admin.site.urls만 예외이다.

path 함수

paht 함수는 URL 패턴을 받아 특정 view를 호출하는 함수이다. 필수 인자로 route, view 2가지를 받는다.

  • route : URL패턴을 가진 문자열이다. request를 처리할 때 Django는 요청된 URL패턴과 urlpatterns 리스트에서 순서대로 route URL패턴과 일치하는 패턴을 찾을때까지 비교한다. 그리고 각 패턴들은 GET 혹은 POST의 매개변수(?page=3), 도메인 이름은 검색하지 않는다.(www.example.com) 위의 예시에선 URLconf는 “www.example.com/coco_app/" 중 “coco_app/“ 부분만 바라보게 된다.
  • view : 앞에서 일치하는 URL 패턴을 찾으면 HttpRequest 객체를 첫번째 인수로 하고, 경로로부터 <캡쳐된?> 값을 키워드 인수로하여 특정한 view 함수를 호출한다.
